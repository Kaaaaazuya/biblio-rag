# 0003. メッセージングに SQS を採用

- 状態: Accepted
- 日付: 2026-06-16
- 関連: [design.md](../design.md) §3

## コンテキスト

各フェーズは処理時間がバラバラで、直列に繋ぐと遅い相がボトルネックになる。フェーズ間を疎結合化し、
ワーカー障害時にもジョブを失わない仕組みが要る。

## 決定

フェーズ間連携に **Amazon SQS** を採用する。`extract` / `chunk` / `embed` の3キュー＋各 DLQ。

## 理由

- 1冊→1ワーカーの **Point-to-Point** 配信に合致。
- メッセージ保持・自動リトライ（可視性タイムアウト）・DLQ が標準装備。
- フェーズごとに独立スケール可能。LocalStack でローカル再現が容易。

## 却下した代替

- **SNS**: 1対多ファンアウト向けで、メッセージ保持がない。本用途は1対1。
- **EventBridge**: ルーティングの柔軟性が不要で、単純なキュー用途では SNS+SQS より高コストになりうる。

## 結果

- 良い点: 疎結合・障害耐性・リトライ/DLQ が容易。
- 悪い点: キューと DLQ の運用（監視・再投入）が必要。
- 注記: フェーズ順序の厳密制御や複雑なリトライが必要になれば Step Functions を検討（[ADR 0004](0004-fargate-zero-scale.md) 代替案C）。
