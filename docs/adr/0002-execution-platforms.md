# 0002. 実行基盤の使い分け（① Fargate / ②③ Lambda）

- 状態: Accepted
- 日付: 2026-06-16
- 関連: [design.md](../design.md) §2, §10、[ADR 0004](0004-fargate-zero-scale.md)

## コンテキスト

各フェーズで処理特性が異なる。① 抽出は見出しのフォントサイズ相対判定に書籍全体が必要で、
1冊1プロセスが素直・処理が重い。②③ は1単位が数秒〜数十秒で状態を持たない。

## 決定

- **① 抽出 = ECS Fargate**。実行時間制限なし・大きめメモリ。書籍全体を1プロセスで扱える。
- **②③ チャンク・埋め込み = Lambda（SQS トリガー）**。アイドル課金なし・配線がシンプル。

## 理由

- Lambda の15分上限と、フォントサイズ相対判定に書籍全体が要る点から、① を Lambda に載せるのは不向き。
- ②③ は短時間・状態不要で SQS が直接トリガーでき、AWS 公式の RAG 取り込みリファレンスでも定番。

## 結果

- 良い点: 各層を処理特性に合った基盤で動かせる。②③ はアイドル¥0。
- 悪い点: 基盤が2種類になり構成・デプロイがやや複雑。
- 代替: 全部 Fargate（②③ のアイドル課金・配線が増える）/ 全部 Lambda（① が15分上限に抵触）。
