# 0007. ② チャンク戦略（ヒューリスティック採用・AI は将来差し替え）

- 状態: Accepted
- 日付: 2026-06-16
- 関連: T3（② チャンク）、[design.md](../design.md)（配置予定）

## コンテキスト

② チャンク層で、`normalized/*.md` を検索単位（チャンク）に分割する方式を決める必要がある。
分割の賢さには3段階の選択肢がある。

1. **ヒューリスティック**: 文字数ベース（400〜600字＋overlap 80字）＋句点「。」優先＋
   ①が復元した見出し階層の境界を尊重し、見出しを prefix として付与。
2. **埋め込みベースの semantic chunking**: bge-m3 の埋め込みで話題の切れ目を検出して分割。
   既に持っている埋め込み器を再利用するため追加コストはほぼゼロ。
3. **LLM 判定**: LLM に分割点や構造を判断させる。品質は最も柔軟だが、課金・遅延・
   非決定性が増える。

本プロジェクトの性質: 対象はテキスト埋め込み済み・フォント階層が明確な日本語 PDF。
開発はローカル完結（無料）で、`chunks/*.jsonl` を「再実行できる正本」とする思想。
決定性・再現性・低コストを重視する。

## 決定

**MVP ではヒューリスティック（方式1）を採用する。** AI（方式2・3）は採用しない。

ただし将来の差し替えに備え、**`Chunker` を抽象インターフェースとして定義**し、
ヒューリスティック実装を既定とする（`Embedder` / `VectorStore` と同じ思想）。

## 理由

- ①が見出し階層という強い構造情報を既に復元しており、「見出し境界を尊重＋セクション内を
  文字数＋句点で分割」で MVP は破綻しにくい。
- ヒューリスティックは決定的で無料・デバッグ容易。正本を何度も再生成する運用に合う。
- AI は非決定性・依存・（本番では）課金を持ち込む。対象 PDF が素直な以上、MVP 段階で
  入れる便益が薄く、過剰実装になる。

## 結果

- 良い点: 速く動く・再現可能・コスト¥0。MVP の縦串を最短で通せる。
- 悪い点: レイアウトが乱れた書籍や表・図の多い書籍では分割品質が落ちうる。
- 緩和策: `Chunker` 抽象により、後から実装だけ差し替え可能。

## 将来の見直しトリガ

実際の書籍で品質に不満が出たら、コストの低い順に検討する。

1. **埋め込みベースの semantic chunking**（bge-m3 を再利用・追加コストほぼゼロ）。
2. それでも不足する場合に **LLM 判定**（`Chunker` 実装を差し替え）。
3. **LLM は "評価役" として先に使う**のが費用対効果が高い: 取り込み結果（チャンク境界・
   見出し検出）をオフラインでスコアリングし、ホットパスには入れない。

抽出①についても同様に、フォント情報が乏しい PDF・スキャン PDF（OCR/Vision）が必要に
なった時点で、AI 支援を別途検討する（本 ADR のスコープ外）。
